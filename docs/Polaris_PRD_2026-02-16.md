# Polaris PRD (Product Requirements Document)

## 1. 문서 정보
- 문서명: Polaris PRD
- 버전: v1.0
- 작성일: 2026-02-16
- 기준 코드: /Users/neo/polaris (main, 최신 로컬 커밋 기준)
- 대상 플랫폼: iOS, Android, Web(정적 배포)

## 2. 제품 한 줄 정의
Polaris는 "목표-실행-정리"를 하나의 흐름으로 연결하는 개인 생산성 앱이다.

## 3. 배경 및 문제 정의
사용자는 목표, 할일, 일정, 메모를 여러 도구에 분산 저장해 다음 문제가 반복된다.
1. 오늘 무엇을 해야 하는지 결정 피로가 큼.
2. 목표와 실제 행동(할일/일정)이 분리되어 진척이 끊김.
3. 상황에 맞는 우선순위 조정이 어렵고 미루기 쉬움.
4. 사용자의 실제 업무 형태(예: 콘텐츠 제작 정산)를 반영한 도구가 부족함.

Polaris는 위 문제를 해결하기 위해 모듈형 구조, 일정 연동, AI 코칭, 업무 특화 모듈을 결합한다.

## 4. 제품 목표
### 4.1 비즈니스 목표
1. 데모 사용자 전환률 개선(데모 -> 회원가입 -> 활성 사용자).
2. 주간 활성 사용자(WAU) 및 4주 리텐션 증가.
3. 핵심 모듈(할일/목표/메모/정산)의 다중 모듈 사용률 확대.

### 4.2 사용자 목표
1. 1분 내 오늘의 우선순위를 정한다.
2. 목표를 월/연 단위로 관리하고 진행률을 확인한다.
3. 일정과 실행 항목을 한 화면 흐름에서 관리한다.
4. 막히는 순간 Compass로 대화하고 즉시 실행 항목으로 전환한다.

### 4.3 비목표 (현재 범위 제외)
1. 팀 협업(공동 편집/권한 관리).
2. 복잡한 프로젝트 간트/의존성 관리.
3. 금융기관 API 직접 연동(자산 자동 수집).

## 5. 대상 사용자 및 페르소나
### 5.1 핵심 페르소나 A: 1인 창작자/프리랜서
- 니즈: 일정, 작업 진행, 정산까지 한 곳에서 관리.
- 문제: 촬영/편집/납품/입금 상태 추적이 번거롭고 누락 위험이 큼.
- 성공 기준: 미수금/납기 누락이 줄고 월별 매출 흐름을 빠르게 파악.

### 5.2 핵심 페르소나 B: 목표 지향 개인 사용자
- 니즈: 목표를 실제 행동으로 쪼개고 유지.
- 문제: 계획은 세우지만 실행 전환과 유지가 어려움.
- 성공 기준: 주간 단위로 목표 진척이 수치로 보이고 실행이 이어짐.

## 6. 핵심 가치 제안
1. 모듈형 개인화: 하단 탭(1,2,4 슬롯) 모듈을 사용자 맥락에 맞게 교체.
2. 실행 중심 흐름: 할일/목표/메모/일정이 분절되지 않고 연결됨.
3. Compass 코칭: 대화형 조언 + 제안 할일 원클릭 반영.
4. 현실 업무 대응: 영상 발주/정산 워크플로우를 제품 내 기본 제공.
5. 낮은 진입 장벽: 데모 모드 즉시 체험 + 필요 시점 전환 유도.

## 7. 현재 기능 범위 (As-Is)
### 7.1 인증 및 계정
1. 이메일 회원가입/로그인.
2. 카카오 로그인.
3. iOS Apple 로그인.
4. 데모 모드 진입/이탈.
5. 계정 삭제(데이터 삭제 + Auth 삭제 + 카카오 unlink 처리).

### 7.2 핵심 생산성 모듈
1. 할일(Todo)
- 생성/수정/삭제/완료/중요표시.
- 카테고리 관리 및 순서 변경(로컬 순서 반영).
- 홈 화면에서 오늘 실행 중심으로 노출.

2. 목표(Goals)
- 연도별 목표 관리.
- 월간 상태형/퍼센트형 목표 지원.
- 월별 진행값 입력 및 달성률 자동 계산.
- 드래그 정렬 및 월간 평가.
- 연간 목표 텍스트 기록.

3. 메모(Memo)
- 리치 텍스트 메모 작성/편집/삭제.
- 카테고리/컬러, 즐겨찾기, 필터, 정렬.

4. 일정(Schedule)
- Google Calendar OAuth 연동.
- 캘린더 선택 동기화, 일정 조회/추가/수정/삭제.
- 주간 날짜 탐색 및 일자별 일정 표시.

5. 컴파스(Compass)
- Gemini 기반 대화형 코칭.
- 사용자 컨텍스트(할일/메모/목표) 기반 응답.
- 제안 할일 카드 + 즉시 추가/중요표시.
- 대화 기록 저장(로그인 사용자 서버 저장, 데모 로컬 저장).

6. 영상 발주 관리(Settlement)
- 작업 전/작업 중/납품 완료/입금 완료 상태 관리.
- 촬영/편집/납품/입금 예정일, 단가, 거래처 관리.
- 상태 전이, 워크플로우 체크리스트, 월별 정산 관점 제공.

### 7.3 개인화 및 탐색
1. 하단 탭 슬롯(1,2,4) 모듈 교체.
2. 전체 모듈 카탈로그(출시/준비중 구분).
3. 더보기 영역에서 모듈 진입.

### 7.4 운영/분석
1. PostHog 이벤트/스크린 트래킹.
2. Slack Webhook 기반 주요 행동 로그/피드백 알림.

## 8. 단기 로드맵 범위 (To-Be)
### 8.1 우선 구현 대상
1. 가계부 모듈 정식화(현재 샘플 상태).
2. 습관 모듈 정식화(현재 샘플 상태).
3. Compass 품질 고도화(개인화 강도, 제안 정확도, 톤 안정화).
4. 일정 통합 UX 개선(외부 캘린더 구독 시나리오 정리).

### 8.2 후속 검토 대상
1. 구독 관리, 몰입 타이머, 비전보드, 장기 계획 모듈.
2. 푸시 알림/리마인더 정교화.

## 9. 사용자 여정
### 9.1 온보딩
1. 앱 진입 -> 데모 체험 또는 로그인/회원가입 선택.
2. 데모에서 핵심 화면 경험.
3. 액션 시점 넛지(회원가입 유도) -> 전환.

### 9.2 일일 사용
1. 홈에서 날짜/확언/디데이 확인.
2. 오늘의 할일 정리 및 중요 항목 설정.
3. 일정 확인 및 필요한 이벤트 추가.
4. 목표/메모 업데이트.
5. 막힐 때 Compass 대화 후 제안을 바로 실행.

### 9.3 창작자 정산 시나리오
1. 신규 발주 건 등록.
2. 촬영/편집 단계 진행 체크.
3. 납품 완료 후 입금 대기 관리.
4. 입금 완료 처리 및 월별 흐름 점검.

## 10. 기능 요구사항 (Functional Requirements)
### 10.1 인증/세션
- FR-AUTH-001: 이메일/카카오/Apple 로그인 지원.
- FR-AUTH-002: 로그인 상태는 플랫폼별 저장소에 유지.
- FR-AUTH-003: 데모 모드에서 쓰기 액션은 제한하고 전환 넛지 제공.
- FR-AUTH-004: 계정 삭제 시 사용자 데이터와 인증 계정을 모두 제거.

### 10.2 생산성 코어
- FR-CORE-001: Todo CRUD, 완료/중요/카테고리 변경 가능.
- FR-CORE-002: Goals는 연도 단위 조회 및 타입별(월간/퍼센트) 관리 가능.
- FR-CORE-003: Memo는 리치 텍스트와 카테고리 필터를 지원.
- FR-CORE-004: 홈 화면은 실행 우선 정보(할일 중심)를 빠르게 보여야 함.

### 10.3 일정
- FR-SCH-001: Google OAuth로 캘린더 계정을 연결/해제 가능.
- FR-SCH-002: 선택 캘린더의 이벤트를 동기화하고 일자별 조회 가능.
- FR-SCH-003: 앱 내 이벤트 생성/수정/삭제 지원.
- FR-SCH-004: 동기화 실패 시 사용자에게 재시도 가능한 오류 상태를 제공.

### 10.4 Compass
- FR-CMP-001: 사용자 메시지 기반 대화 응답 생성.
- FR-CMP-002: 응답은 할일 제안 목록(선택적)을 포함 가능.
- FR-CMP-003: 제안 항목을 기존 할일과 매칭해 중복 생성 최소화.
- FR-CMP-004: 제안 수락 시 즉시 Todo 반영(신규 생성 또는 중요표시).
- FR-CMP-005: 대화 이력은 사용자별로 저장/재로딩 가능.

### 10.5 정산
- FR-STL-001: 발주 작업 CRUD 지원.
- FR-STL-002: 작업 상태를 표준 단계로 관리.
- FR-STL-003: 날짜/단가/거래처/작업 타입 필드 지원.
- FR-STL-004: 상태 전환 이벤트 및 기록 반영.

### 10.6 모듈 개인화
- FR-MOD-001: 사용자별 모듈 활성화/비활성화 상태 저장.
- FR-MOD-002: 탭 슬롯 1/2/4 모듈 교체 가능(중복 방지).
- FR-MOD-003: 사용 빈도 기반 최근 모듈 우선 노출.

## 11. 비기능 요구사항 (Non-Functional Requirements)
### 11.1 성능
1. 핵심 화면 최초 진입 시 로딩 상태를 명확히 노출.
2. 데이터 변경은 낙관적 업데이트를 기본 적용해 체감 지연 최소화.
3. 캘린더 데이터는 캐시를 활용해 재진입 속도 확보.

### 11.2 신뢰성
1. Supabase Realtime 기반 동기화 실패 시 재조회 가능.
2. 외부 API 실패(Google/Gemini/Kakao) 시 폴백 메시지 제공.

### 11.3 보안/프라이버시
1. 사용자 데이터는 RLS 정책 기반으로 사용자 단위 격리.
2. OAuth 토큰은 사용자별 저장 및 만료 갱신 처리.
3. 계정 삭제 요청 시 데이터 영구 삭제 보장.

### 11.4 운영성
1. 주요 사용자 행동은 Analytics(PostHog)로 추적.
2. 핵심 이벤트는 Slack 로그로 운영 가시성 확보.

## 15. 리스크 및 대응
1. 외부 API 의존 리스크(Google/Gemini)
- 대응: 캐시/재시도/실패 메시지/기능별 폴백.

2. 데모-실사용 간 기대 불일치

3. 모듈 확장 시 복잡도 증가
- 대응: 모듈 인터페이스 표준화, 슬롯 규칙 유지, 분석 기반 우선순위.

4. 개인정보 처리 부담 증가
- 대응: 최소 수집 원칙, 삭제 기능 신뢰성 강화, 정책 문서 최신화.

## 16. 오픈 이슈
1. 캘린더 외부 구독(ICS) UX를 Google 흐름과 어떻게 통합할지.
2. Compass의 사용자별 톤/깊이 개인화 옵션 노출 수준.
3. 정산 모듈의 월별 리포트/내보내기 요구 우선순위.
4. 웹 경험을 모바일 프레임 중심으로 유지할지, 데스크톱 전용 레이아웃을 확장할지.

## 17. 부록: 구현 근거 요약
- 모듈 카탈로그 및 슬롯 기반 탭 구조가 구현되어 있음.
- Supabase 기반 실시간 동기화/권한 정책(RLS) 구조가 적용되어 있음.
- Google Calendar OAuth + 이벤트 CRUD + 토큰 갱신 흐름이 구현되어 있음.
- Gemini 기반 Compass 대화 및 제안 할일 반영 흐름이 구현되어 있음.
- Settlement(영상 발주/입금 관리) 전용 도메인 모듈이 구현되어 있음.
